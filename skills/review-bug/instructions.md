# Bug 审查指令

当用户调用 `/review-bug` 时，执行以下流程：

## 1. 确定审查范围

- 无参数：审查整个 `src/` 目录
- 有参数：审查用户指定的目录或文件
- 跳过 `ui/` 目录下的基础组件（shadcn/ui 等第三方组件）

## 2. 扫描阶段

使用 Agent 工具并行扫描，按以下维度检查：

### 必查项
- [ ] 所有页面的 `mx-`、`px-`、`w-` 类是否在 360px 下溢出
- [ ] 所有颜色类是否有 `dark:` 变体
- [ ] 所有 `disabled` 条件是否逻辑正确
- [ ] 所有 `useEffect` 是否有正确的 cleanup
- [ ] 所有列表渲染是否有唯一 `key`
- [ ] 所有图片是否有 `alt` 属性
- [ ] 所有 `<a>` 标签是否有 `target="_blank"` + `rel="noopener noreferrer"`
- [ ] TypeScript 类型断言（`as`）是否安全
- [ ] 文件编码是否正常（无乱码）

### 选查项（根据项目技术栈）
- [ ] Next.js Script 组件 strategy 是否正确
- [ ] Tailwind 动态类名是否会被 JIT 扫描到
- [ ] next/image 的 sizes 属性是否合理
- [ ] 结构化数据 (JSON-LD) 是否正确

## 3. 验证阶段

对每个可疑问题：
1. 确认是否为实际 bug（非设计偏好）
2. 确认影响范围（单页面 vs 全站）
3. 确认严重程度

## 4. 输出报告

按严重程度排序，输出 Top 5 Bug（或用户指定数量）。

每个 Bug 必须包含：
- 文件路径和行号
- 问题描述
- 影响说明
- 具体修复方案（含代码）

## 5. 后续操作

报告完成后，询问用户：
- 是否需要修复这些 bug
- 是否需要更详细的某个 bug 分析
- 是否需要扩展审查范围
